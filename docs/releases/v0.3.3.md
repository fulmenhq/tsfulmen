# Release v0.3.3 (2026-06-17)

**Release Type**: Patch — compile-safety ergonomics

## Overview

Two purely-additive API options that make it easier to consume tsfulmen from a `bun --compile`
single-file binary, surfaced by `forge-workhorse-tuvan` while building its compiled binaries. A
single-file binary has no `.fulmen/app.yaml` on disk and no filesystem-backed schema/config assets,
so the previous APIs forced consumers into per-build workarounds (temp-files, FS injection). v0.3.3
makes the two most common of those first-class.

This is the small, low-risk fast-follow ahead of the larger **v0.4.0** work, which will embed
tsfulmen's SSOT assets (schemas, metaschemas, foundry catalogs) so all asset loads are
filesystem-independent and standalone `serve` works in a compiled binary.

No breaking changes. The existing exported `LoadConfigOptions` shape is unchanged, the public API is
otherwise additive, and the Node engine floor is unchanged (`>=22.12.0`).

## What Changed

### Added

- **`skipValidation` on `registerEmbeddedIdentity(data, options?)`** — symmetric with the existing
  `loadIdentity({ skipValidation })`. The schema registry is filesystem-backed and absent inside a
  `bun --compile` binary, so the prior `registerEmbeddedIdentity(data)` (which always validated)
  threw `SchemaValidationError: Schema not found …app-identity`. With `{ skipValidation: true }`, a
  consumer can register a build-time, CI-validated identity without touching the registry. Parse /
  first-wins / deep-freeze semantics are unchanged; only schema validation is gated. New
  `RegisterEmbeddedIdentityOptions` type is exported from `appidentity`.
- **Inline `defaults` / `schema` on `loadConfig`** — provide a pre-parsed `defaults` object and/or an
  inline `schema` string instead of file paths. This removes the embed → temp-file → pass-path dance
  for consumers that embed their config at build time. Exposed via a new `LoadInlineConfigOptions`
  shape and a `loadConfig` overload. `defaults` is `structuredClone`'d so a caller's embedded object
  is never mutated by the merge.

### Changed

- **`loadConfig` options split into path-based and inline shapes** (additive, semver-safe). A shared
  `BaseLoadConfigOptions` holds the common fields; `LoadConfigOptions` keeps `defaultsPath: string`
  (required) exactly as in v0.3.2; the new `LoadInlineConfigOptions` carries `defaults` (required).
  The two are mutually exclusive at the type level (`?: never`), and `loadConfig` is overloaded over
  them. Because the exported `LoadConfigOptions` is byte-for-byte unchanged, strict TypeScript
  consumers reading `options.defaultsPath` as `string` are unaffected on a `^0.3.2` → `0.3.3` update.
- **`ConfigMetadata` diagnostics** — adds optional `defaultsSource: "path" | "inline"` and
  `schema.source: "path" | "inline" | null`. `defaultsPath` stays typed `string` (inline reports
  `""`). Also fixes a reporting bug: when both inline `schema` and `schemaPath` were supplied, inline
  won validation but `schema.path` still reported the unused `schemaPath`; inline now correctly
  reports `path: null` with `source: "inline"`.

## Consumer Impact

**Additive.** No removals, no signature changes to existing call shapes. Path-based `loadConfig`
callers and existing `registerEmbeddedIdentity` callers need no changes. New options are opt-in and
primarily benefit consumers compiling single-file binaries.

## Testing & Validation

- `make check-all` — typecheck, lint, full suite (**2142 passed / 16 skipped**); app-identity and
  schema-export parity pass.
- 11 new tests: `skipValidation` registration paths (valid YAML, schema-invalid object accepted when
  skipped, malformed YAML still rejected, freeze + first-wins preserved); inline `defaults`/`schema`
  loading, no-mutation, inline-wins-over-`schemaPath` precedence, and the missing-source guard.
- `make verify-artifacts` / `verify-local-install` / `verify-compile-smoke` — green (see release PR).

## Follow-ups

- **v0.4.0 (epic):** embed tsfulmen SSOT assets via an `AssetResolver` layer so all loads are
  filesystem-independent in a compiled binary — unblocks standalone `serve`. Coordinated in
  `#release-tsfulmen-v040`; shaping in `planning/tsfulmen/v040-shaping.md`.
- Downstream: `forge-workhorse-tuvan` adopts these two options in its compiled-binary rebuild
  (maps 1:1 to the identity + config workarounds it currently carries).
