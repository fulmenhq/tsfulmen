# Release v0.4.0 (2026-06-24)

**Release Type**: Minor — compile-safe SSOT asset embedding

## Overview

v0.4.0 makes tsfulmen's bundled SSOT assets — JSON Schemas, JSON-Schema metaschemas,
foundry catalogs, and the telemetry taxonomy — resolve **without the filesystem**, so
the library works fully inside a `bun build --compile` single-file binary. This unblocks
the last gap surfaced by `forge-workhorse-tuvan`: **standalone `serve`**, plus schema
discovery and config validation, now work in a compiled binary.

The work was driven by an architectural class of bug: tsfulmen loaded SSOT assets from
the filesystem (via `import.meta.url`/`__dirname`-relative `readFile`, and a runtime
`fast-glob` walk in the schema registry), none of which survive `--compile` (no file
tree on disk). v0.4.0 introduces a central `AssetResolver` with filesystem and embedded
backends and routes every SSOT load through it.

No public API removals; the new surface is additive. Node engine floor unchanged
(`>=22.12.0`).

## What Changed

### Added

- **`@fulmenhq/tsfulmen/assets` — the `AssetResolver` layer.**
  - `FsAssetResolver` (reads the on-disk `schemas/crucible-ts` + `config/crucible-ts`
    trees) and `EmbeddedAssetResolver` (reads build-embedded generated modules).
  - `resolveAssets()` / `getAssetResolver()` choose the backend via
    `TSFULMEN_ASSET_MODE` (`auto` | `fs` | `embedded`). Default `auto`: filesystem when
    the asset tree is present (npm / `node dist` / source), embedded otherwise
    (compiled binary).
  - Logical paths are package-root-relative POSIX and validated against traversal /
    absolute / non-POSIX inputs and restricted to the SSOT namespaces.
- **Embedded SSOT asset modules** (`src/assets/generated/`, checked in): per-domain
  manifests (metaschema, schemas, foundry catalogs, taxonomy) emitted by
  `scripts/embed-assets.ts`. Regenerate with `make embed-assets` after `make sync-ssot`;
  `make verify-embedded-assets` (in `check-all`) fails on drift vs. the on-disk trees.
- **Compile-safety CI guards:**
  - `make verify-embedded-compile` — compiles `bun build --compile` fixtures and runs
    them from a temp cwd with no asset tree, asserting embedded read + enumeration AND
    standalone `serve` (signals catalog load + Prometheus metrics server bind/shutdown,
    the real `serveCommand` lifecycle). Enforced in the CI build job.
  - `make verify-dist-asset-partition` — asserts public entry files stay lean, the
    schema corpus lives in exactly one shared chunk, every domain is present, with a
    size report. Wired into `make build`.

### Changed

- **Every SSOT asset load routes through the `AssetResolver`**: schema registry
  discovery (`resolver.list` — no FS walk), validator metaschemas / vocabulary /
  `compileSchemaById` / cross-tree `$ref` resolution, foundry catalogs + signals
  catalog, and telemetry taxonomy. `SchemaMetadata.path` is now a logical
  (resolver-relative) path read back through the resolver. This also fixes latent
  cwd/dist-layout fragility in the previous `__dirname`-relative loaders.
- **`tsup splitting: true`** — the embedded corpus is deduped into one shared chunk
  instead of duplicated into every validation-using entry (dist ~3.7 MB vs ~24 MB),
  and public entry files stay lean. Published `dist/` now includes `chunk-*.js`
  (packed automatically via `files: ["dist"]`).
- **Crucible SSOT `v0.4.14 → v0.4.15`** — fixes cross-schema `$ref`/`$id` vs.
  file-layout inconsistencies (observability/logging `$id` now version-as-directory;
  module-manifest relative-ref correction) that the resolver repoint surfaced.

### Fixed

- **Standalone `serve`, schema discovery, and config validation now work inside a
  `bun build --compile` single-file binary** — the foundry signals catalog (the
  original blocker) and the rest of the SSOT assets resolve from embedded modules
  instead of an absent on-disk tree.

## Consumer Impact

- **Additive API.** New `@fulmenhq/tsfulmen/assets` export and `TSFULMEN_ASSET_MODE`
  env var. No existing signatures removed; `node dist` / npm behavior is unchanged
  (filesystem mode by default).
- **`bun --compile` consumers** (e.g. CDRL workhorses): the full SDK — including
  `serve` and schema validation — now works in a single-file binary with no asset
  tree on disk. See `docs/development/compile-safe-assets.md`.
- **Published package** gains shared `dist/chunk-*.js` files; subpath resolution is
  unchanged from a consumer's perspective.

## Testing & Validation

- `make check-all` — typecheck, lint, full suite (**2194+ tests**), license-audit,
  schema-export + app-identity + signals parity, embedded-asset drift.
- `make build` (+ `verify-dist-asset-partition`), `make verify-artifacts`,
  `make verify-local-install`, `make verify-compile-smoke`,
  `make verify-embedded-compile` (read/enumerate **and** in-binary `serve`).

## Follow-ups

- Downstream: `forge-workhorse-tuvan` can run standalone `serve` in its compiled
  binaries and drop its `serve`-descope workaround.
- Hooks hygiene: the pre-push `goneat assess` timeout on asset-heavy branches.
